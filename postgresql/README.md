# PostgreSQL 18.3

Standalone PostgreSQL 18.3 via Docker Compose. Designed for Coolify's Docker Compose deployment mode.

## Quick Start

1. Copy `.env.example` to `.env` and set your password:
   ```bash
   cp .env.example .env
   ```
2. Edit `.env` — set `POSTGRES_PASSWORD` to a strong value
3. Deploy:
   ```bash
   docker compose up -d
   ```

## Coolify Setup

1. Create a new resource in Coolify
2. Set the **Build Pack** to **Docker Compose**
3. Set the **Docker Compose Location** to `postgresql/docker-compose.yml`
4. Deploy — password is auto-generated via Coolify's `SERVICE_PASSWORD_POSTGRES` magic variable

> **Note:** No manual environment variables or storage configuration needed in Coolify. The password is auto-generated on first deploy and persists across redeployments. Named volumes are automatically detected and managed.

## Environment Variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `SERVICE_PASSWORD_POSTGRES` | Auto (Coolify) | Auto-generated | PostgreSQL superuser password (Coolify magic variable) |
| `POSTGRES_USER` | No | `postgres` | PostgreSQL superuser name |

> **Note:** For local development, copy `.env.example` to `.env` and set `POSTGRES_PASSWORD` manually. Optionally set `POSTGRES_DB` to create an additional database on first init.

## Data Persistence

Data is stored in the `pgdata` named volume (`/var/lib/postgresql`). The database is initialized only on first run — subsequent restarts reuse existing data.

> **Note:** PostgreSQL 18+ changed its data directory structure. The volume mount point is `/var/lib/postgresql` (not `/var/lib/postgresql/data`). See [docker-library/postgres#1259](https://github.com/docker-library/postgres/pull/1259) for details.

## Connecting Your Services

Each Docker Compose stack in Coolify gets its own isolated network. To connect your app (a separate Coolify resource) to this PostgreSQL service, you need to enable Coolify's **Connect to Predefined Network** feature.

### Step 1: Enable cross-resource networking

1. Go to your **PostgreSQL resource** in Coolify → **General** → enable **Connect to Predefined Network**
2. Go to your **app resource** in Coolify → **General** → enable **Connect to Predefined Network**

This connects both stacks to the shared destination Docker network.

### Step 2: Determine the hostname

When **Connect to Predefined Network** is enabled, Coolify renames containers to prevent collisions. The hostname is **not** `postgresql` — it becomes:

```
postgresql-<resource-uuid>
```

Find the resource UUID in the Coolify URL when viewing the PostgreSQL resource, or in the resource's **General** tab.

### Connection details

| Parameter | Value |
|---|---|
| **Host** | `postgresql-<resource-uuid>` (see above) |
| **Port** | `5432` |
| **User** | `postgres` |
| **Password** | Value of `SERVICE_PASSWORD_POSTGRES` (auto-generated by Coolify) |
| **Database** | `postgres` |

### Connection string

```
postgresql://postgres:<password>@postgresql-<resource-uuid>:5432/postgres
```

### Step 3: Share the password with your app

Use Coolify's **shared environment variables** to avoid hardcoding the password:

1. Go to your **Environment** (under the Project) → gear icon → add a shared variable, e.g. `DB_PASSWORD`
2. Set its value to the generated password (visible in the PostgreSQL resource's environment tab)
3. In your app's environment variables, reference it: `DATABASE_PASSWORD={{environment.DB_PASSWORD}}`

Alternatively, manually copy the password from the PostgreSQL resource's environment tab into your app's environment variables.

### Example app environment variables

```
DATABASE_HOST=postgresql-<resource-uuid>
DATABASE_PORT=5432
DATABASE_USER=postgres
DATABASE_PASSWORD={{environment.DB_PASSWORD}}
DATABASE_URL=postgresql://postgres:{{environment.DB_PASSWORD}}@postgresql-<resource-uuid>:5432/postgres
```

### Important caveats

- **Internal DNS changes**: Enabling "Connect to Predefined Network" changes how Docker DNS works. Within the same compose stack, use `postgresql-<uuid>` instead of just `postgresql`.
- **No host port mapping**: No ports are exposed to the host. All communication happens over Docker's internal network.
- **Same stack alternative**: If your app and PostgreSQL are always deployed together, put them in the same `docker-compose.yml` — then `postgresql:5432` works directly with no extra configuration.
